## 2025-03-01 - Email Header Injection via CRLF in API

**Vulnerability:** The `/api/quote` endpoint used user input (`data.name` and `data.type`) directly in the email `subject` field and `data.email` in the `reply_to` field for the Resend API request. While the application had a `sanitizeInput` function to mitigate XSS (escaping HTML characters like `<`, `>`, `&`, `"`, `'`), it failed to strip out newline characters (`\r` and `\n`). Additionally, `data.email` was not validated to ensure it was a valid email address before being used as a header value.
**Learning:** HTML XSS sanitization does not protect against Email Header Injection. If user input is mapped to email headers (like `Subject` or `Reply-To`), an attacker can inject CRLF (`\r\n`) sequences to add arbitrary headers (like `Bcc`) or alter the email structure, potentially facilitating phishing or spam campaigns.
**Prevention:** Always validate that email address fields actually match an email format. Furthermore, completely strip newline characters (`\r` and `\n`) from any user input that will be interpolated into email headers (e.g., the subject line) before transmitting the payload to an email service provider or SMTP server.
